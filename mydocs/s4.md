# s04 学习笔记 —— Subagents

## 核心思想

> **"Break big tasks down; each subtask gets a clean context"**
>
> 把大任务拆解，每个子任务获得干净的上下文。s04 的核心是 **上下文隔离** —— 子智能体在独立的 messages 里运行，完成后只返回摘要。

## 新增内容（相比 s03）

### 1. 两个 System Prompt

```python
SYSTEM = "You are a coding agent... Use the task tool to delegate exploration or subtasks."
SUBAGENT_SYSTEM = "You are a coding subagent... Complete the given task, then summarize your findings."
```

| | 父智能体 | 子智能体 |
|--|---------|---------|
| 身份 | coding agent | coding subagent |
| 指令 | 用 task 工具委派子任务 | 完成任务并总结发现 |

### 2. CHILD_TOOLS —— 子智能体的工具

```python
CHILD_TOOLS = [bash, read_file, write_file, edit_file]
```

**子智能体没有 `task` 工具！** 防止无限递归派生。

### 3. PARENT_TOOLS —— 父智能体的工具

```python
PARENT_TOOLS = CHILD_TOOLS + [task]
```

父智能体比子智能体多一个 `task` 工具，用于委派子任务。

### 4. run_subagent —— 核心函数

```python
def run_subagent(prompt: str) -> str:
    sub_messages = [{"role": "user", "content": prompt}]  # ← 全新上下文！
    for _ in range(30):  # ← 安全限制，最多30轮
        response = client.messages.create(
            model=MODEL, system=SUBAGENT_SYSTEM, messages=sub_messages,
            tools=CHILD_TOOLS, max_tokens=8000,
        )
        sub_messages.append({"role": "assistant", "content": response.content})
        if response.stop_reason != "tool_use":
            break
        # ... 执行工具，组装 tool_result ...
        sub_messages.append({"role": "user", "content": results})
    # 只返回最终文本，子上下文全部丢弃
    return "".join(b.text for b in response.content if hasattr(b, "text"))
```

**关键点：**

| 代码 | 含义 |
|------|------|
| `sub_messages = [{"role": "user", "content": prompt}]` | 子智能体从零开始，看不到父智能体的历史 |
| `tools=CHILD_TOOLS` | 子智能体没有 task 工具，不能继续派生 |
| `for _ in range(30)` | 最多30轮，防止无限循环 |
| `return "".join(b.text for ...)` | **只返回最终文本摘要，中间过程全部丢弃** |

### 5. agent_loop 的变化

```python
if block.name == "task":
    desc = block.input.get("description", "subtask")
    prompt = block.input.get("prompt", "")
    print(f"> task ({desc}): {prompt[:80]}")
    output = run_subagent(prompt)  # ← 调用子智能体
else:
    handler = TOOL_HANDLERS.get(block.name)
    output = handler(**block.input) ...
```

- 如果工具名是 `task`，调用 `run_subagent(prompt)`
- 子智能体运行结束后，返回的字符串作为 tool_result content
- 父智能体的 messages 里只多一条摘要，**上下文保持干净**

## 交互流程示例

用户：`帮我分析这个项目里所有 Python 文件的复杂度`

**父智能体：**
```
→ 思考：文件很多，委派给子智能体
→ tool_use: task(prompt="分析 agents/ 下所有 .py 文件的圈复杂度...")
```

**子智能体（独立运行，父看不到）：**
```
Round 1: bash → find agents -name "*.py"
Round 2: read_file → s01_agent_loop.py
Round 3: read_file → s02_tool_use.py
...
Round N: end_turn → "分析完成。共12个文件，平均复杂度8.5..."
```

**父智能体收到：**
```
tool_result: "分析完成。共12个文件，平均复杂度8.5，最高s09（23），最低s01（4）。"
```

父智能体的上下文里**只有这一句话**，看不到子智能体中间读了多少文件。

## 常见问题

### Q1：`description` 不是 required 的，为什么模型还是会传真实值？

**因为 `required` 只控制"必须传"，不控制"能不能传"。**

Schema 里 `description` 虽然不在 `required` 里，但模型**可以自主选择传或不传**：

```python
"input_schema": {
    "properties": {
        "prompt": {"type": "string"},           // ← required，必须传
        "description": {"type": "string"}       // ← 不在 required 里，但模型可以自愿传
    },
    "required": ["prompt"]
}
```

模型传不传 `description`，取决于它自己的判断：

| 情况 | 模型行为 | `desc` 的值 |
|------|---------|------------|
| 模型觉得"描述一下有用" | 主动传 `description` | 模型写的真实描述 |
| 模型觉得"没必要描述" | 不传 `description` | `"subtask"`（代码默认值）|

**所以 `block.input.get("description", "subtask")` 的默认值 `"subtask"` 只是一个 fallback：**
- 模型传了 → 用模型传的真实值
- 模型没传 → 用 `"subtask"`

这和人类可选填表单的逻辑一样：不是必填的字段，你填了就用你填的，没填就用系统默认值。

### Q2：为什么 `task` 不和其他工具一样放进 `TOOL_HANDLERS`？

**技术上其实可以放进去，但作者刻意分开写，因为 `task` 和普通工具根本不在一个层级。**

普通工具的处理：直接执行，立即返回结果

```python
run_bash("ls") → "文件列表字符串"      # 一次函数调用
run_read("a.py") → "文件内容字符串"    # 一次函数调用
```

`task` 的处理：启动一个完整的 agent_loop，可能调用多次 API

```python
run_subagent("分析所有py文件...")
  → 内部启动一个完整的 agent_loop
  → 可能调用 10 次 API、读 20 个文件
  → 最后只返回一句摘要
```

**如果强行合并到 `TOOL_HANDLERS`：**

```python
TOOL_HANDLERS = {
    "bash":       lambda **kw: run_bash(kw["command"]),
    ...
    "task":       lambda **kw: run_subagent(kw["prompt"]),  # ← 可以这么写！
}
```

这样写**代码能跑**，但会带来问题：

| 问题 | 说明 |
|------|------|
| `run_subagent` 内部有循环 | 普通工具是"同步执行立刻返回"，task 是"启动另一个 agent_loop"，逻辑差异巨大 |
| 错误处理不一致 | 子智能体内部抛异常需要特殊处理，和普通工具不同 |
| 可读性差 | 一眼看去所有工具都一样，看不出 task 是"元工具" |
| 后续扩展麻烦 | s09 之后子智能体可能要加更多逻辑（如超时、结果过滤），混在 TOOL_HANDLERS 里不好改 |

**分开写的真正原因：架构清晰。**

```python
# 普通工具：直接执行
handler = TOOL_HANDLERS.get(block.name)
output = handler(**block.input)

# 元工具：启动子智能体
if block.name == "task":
    output = run_subagent(prompt)
```

这样一眼就能看出：`task` 是特殊的，它触发的是另一个完整的智能体生命周期。

## 设计哲学

| 要点 | 说明 |
|------|------|
| 上下文隔离 | 子智能体从零开始，不被父历史干扰 |
| 摘要返回 | 只返回最终文本，中间过程丢弃 |
| 文件共享 | 共享文件系统，子可以读写父的文件 |
| 防递归 | 子没有 task 工具，不能继续派生 |
| 父上下文干净 | 父只收到摘要，不被子任务细节污染 |
