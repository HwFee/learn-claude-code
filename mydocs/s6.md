# s06 学习笔记 —— Context Compact

## 核心思想

> **"Context will fill up; you need a way to make room"**
>
> 上下文会填满，你需要腾空间的方法。s06 解决的是：**对话历史太长，messages 列表撑爆的问题。**

---

## 问题背景

对话每轮都在追加 messages：

```
Round 1: messages 长度 ~1k tokens
Round 10: messages 长度 ~15k tokens
Round 30: messages 长度 ~50k tokens  ← 撑爆！
```

不处理的话，要么达到 max_tokens 限制，要么 API 费用爆炸。

---

## 三层压缩管道

```
Every turn:
Tool call result
      ↓
[Layer 1: micro_compact]      ← 静默执行，每轮都跑
  把旧的 tool_result 替换为 "[Previous: used {tool_name}]"
      ↓
[Check: tokens > 50000?]
   no ↓          yes ↓
continue    [Layer 2: auto_compact]
              保存完整对话到 .transcripts/
              让 LLM 总结对话
              把所有 messages 替换为 [summary]
                    ↓
            [Layer 3: compact tool]
              模型调用 compact → 立即触发总结
              和 auto 一样，只是手动触发
```

## 新增内容（相比 s05）

### 1. 常量配置

```python
THRESHOLD = 50000          # auto_compact 触发阈值
TRANSCRIPT_DIR = WORKDIR / ".transcripts"  # 完整对话保存目录
KEEP_RECENT = 3            # micro_compact 保留最近 3 个 tool_result
PRESERVE_RESULT_TOOLS = {"read_file"}  # 这些工具的 result 不被压缩
```

### 2. estimate_tokens —— 粗略估算

```python
def estimate_tokens(messages: list) -> int:
    return len(str(messages)) // 4
```

- 把整个 messages 列表转成字符串，除以 4
- 英文大概 4 字符 = 1 token，这是估算，不需要精确

### 3. Layer 1: micro_compact —— 每轮静默压缩

```python
def micro_compact(messages: list) -> list:
    # 1. 收集所有 tool_result
    tool_results = []
    for msg_idx, msg in enumerate(messages):        # msg_idx 实际上没用上
        if msg["role"] == "user" and isinstance(msg.get("content"), list):
            for part_idx, part in enumerate(msg["content"]):  # part_idx 也没用
                if isinstance(part, dict) and part.get("type") == "tool_result":
                    tool_results.append((msg_idx, part_idx, part))  # 索引冗余
    
    # 2. 建立 tool_use_id → tool_name 映射
    tool_name_map = {block.id: block.name for ...}
    
    # 3. 替换旧的 result（保留最后 KEEP_RECENT 个）
    to_clear = tool_results[:-KEEP_RECENT]
    for _, _, result in to_clear:        # ← msg_idx 和 part_idx 被 _ 忽略
        if len(result["content"]) <= 100:
            continue              # 太短的不用替换
        if tool_name in PRESERVE_RESULT_TOOLS:
            continue              # read_file 结果保留
        result["content"] = f"[Previous: used {tool_name}]"
```

> **注：`msg_idx` 和 `part_idx` 被收集进来但后面没用上。** 因为 `part` 是引用类型，直接改 `result["content"]` 就等于改掉了 `messages` 里的原始对象，不需要通过索引定位。更简洁的写法可以直接 `tool_results.append(part)`，不需要索引。

**替换规则：**

| 条件 | 行为 |
|------|------|
| 最近 3 个 tool_result | **保留** |
| content ≤ 100 字符 | **保留**（太短没必要） |
| tool_name 是 `read_file` | **保留**（文件内容是参考资料） |
| 其他 | 替换为 `"[Previous: used {tool_name}]"` |

**为什么保留 `read_file`？**
- `bash ls` 的结果 → 替换后没关系，模型不需要旧目录列表
- `read_file` 的文件内容 → **不能替换**，否则模型要重新读文件

**效果示例：**

```
压缩前:
  bash → "file1.txt file2.txt file3.txt..." (500 chars)
  bash → "mkdir created" (50 chars, 保留-太短)
  read_file → "import os\n..." (1000 chars, 保留-read_file)
  bash → "Wrote 300 bytes" (80 chars, 保留-太短)
  write_file → "Wrote 500 bytes" (200 chars) → 替换

压缩后:
  bash → "[Previous: used bash]"
  bash → "mkdir created" (50 chars)
  read_file → "import os\n..." (1000 chars)
  bash → "Wrote 300 bytes" (80 chars)
  write_file → "[Previous: used write_file]"
```

### 4. Layer 2: auto_compact —— 自动大压缩

```python
def auto_compact(messages: list) -> list:
    # Step 1: 保存完整对话到磁盘
    TRANSCRIPT_DIR.mkdir(exist_ok=True)
    transcript_path = TRANSCRIPT_DIR / f"transcript_{int(time.time())}.jsonl"
    with open(transcript_path, "w") as f:
        for msg in messages:
            f.write(json.dumps(msg, default=str) + "\n")
    
    # Step 2: 让 LLM 自己总结
    conversation_text = json.dumps(messages, default=str)[-80000:]
    response = client.messages.create(
        model=MODEL,
        messages=[{"role": "user", "content": 
            "Summarize this conversation for continuity...\n\n" + conversation_text}],
        max_tokens=2000,
    )
    summary = next((block.text for block in response.content if hasattr(block, "text")), "")
    
    # Step 3: 替换所有 messages 为总结
    return [
        {"role": "user", "content": f"[Conversation compressed. Transcript: {transcript_path}]\n\n{summary}"},
    ]
```

**三步拆解：**

| 步骤 | 做什么 | 为什么 |
|------|--------|--------|
| 保存 | 把完整对话写成 `.transcripts/transcript_时间戳.jsonl` | 原始记录不丢，人类随时可查 |
| 总结 | 把对话发给 LLM，让它生成摘要 | LLM 最擅长理解自己的输出 |
| 替换 | 把几百条 messages 换成 1 条 summary | 从 50k tokens → ~300 tokens |

**`default=str` 的作用：**
- `json.dumps` 遇到 Block 对象（不可序列化）会报错
- `default=str` 让它自动调用 `str()` 转换

**`[-80000:]` 的作用：**
- 只取最后 80k 字符，防止超出 API 输入限制
- 早期轮次的信息相对不重要（已经被 micro_compact 处理过）

**压缩效果：**
- 压缩前：50k tokens（几百条 messages）
- 压缩后：~300 tokens（1 条 summary）
- **压缩比约 150:1**

### 5. Layer 3: compact tool —— 手动压缩

```python
{"name": "compact", "description": "Trigger manual conversation compression.",
 "input_schema": {"type": "object", "properties": {"focus": {"type": "string"}}}}
```

```python
if block.name == "compact":
    manual_compact = True
    output = "Compressing..."
...
if manual_compact:
    print("[manual compact]")
    messages[:] = auto_compact(messages)
    return
```

- 新增 `compact` 工具，模型可以主动调用
- 调用后和 auto_compact 逻辑完全一样
- 压缩后 `return`，结束当前 agent_loop

**为什么需要手动？**
- auto_compact 只有超 50k 才触发
- 但模型可能觉得"现在上下文够乱了，我想主动整理"
- 或者人类提示"请压缩对话"

## agent_loop 的整合

```python
def agent_loop(messages: list):
    while True:
        micro_compact(messages)                    # Layer 1: 每轮都执行
        
        if estimate_tokens(messages) > THRESHOLD:  # Layer 2: 超阈值触发
            messages[:] = auto_compact(messages)
        
        response = client.messages.create(...)     # 正常 API 调用
        messages.append({"role": "assistant", "content": response.content})
        ...
        
        if manual_compact:                         # Layer 3: 手动触发
            messages[:] = auto_compact(messages)
            return
```

**执行顺序：**

| 步骤 | 层级 | 触发条件 |
|------|------|---------|
| 1. micro_compact | Layer 1 | **每轮** |
| 2. 检查 tokens > 50k | Layer 2 | 超过阈值 |
| 3. API 调用 | — | — |
| 4. 执行工具 | — | — |
| 5. 检查 compact | Layer 3 | 模型调用了 compact |

## 设计哲学

| 要点 | 说明 |
|------|------|
| 三层渐进 | micro（每轮）→ auto（超阈值）→ manual（模型主动） |
| 有策略地遗忘 | 保留 read_file、保留最近3个、保留短结果 |
| 原始记录不丢 | 完整对话保存到 `.transcripts/`，随时可查 |
| LLM 自己总结 | 用模型理解模型输出，比算法提取更准确 |
| 无限会话 | 压缩后可以无限循环，理论上会话永不结束 |

## 补充知识点

### Python 引用传递 vs C++ 值传递

`s06` 的 `micro_compact` 里，`result["content"] = "..."` 会**就地修改** `messages` 里的原始数据。这是因为 Python 默认对可变对象（list/dict）使用引用传递。

#### Python vs C++ 对比

| | Python | C++ |
|--|--------|-----|
| **默认行为** | 传引用（修改会影响原始数据） | 传值（修改的是副本） |
| 可变对象（list/dict） | 引用传递 | 值传递（复制整个对象） |
| 不可变对象（int/str） | 值语义（重新赋值创建新对象） | 值传递 |

#### Python 不想用引用怎么办？

```python
import copy

# 浅拷贝（只拷贝最外层，内层对象还是引用）
new_list = old_list.copy()
new_dict = old_dict.copy()

# 深拷贝（递归拷贝所有层级，完全独立）
new_messages = copy.deepcopy(messages)
```

#### hasattr vs isinstance 的选择

`s06` 这行代码：

```python
summary = next((block.text for block in response.content if hasattr(block, "text")), "")
```

`hasattr(block, "text")` 会匹配 **TextBlock 和 ThinkingBlock**（两者都有 `.text` 属性）。

如果模型返回了 ThinkingBlock，代码会拿到**内部思考内容**当总结，这是 bug。

**更严谨的写法：**

```python
from anthropic.types import TextBlock

# 只取 TextBlock，跳过 ThinkingBlock
summary = next((block.text for block in response.content 
                if isinstance(block, TextBlock)), "")
```

| 写法 | TextBlock | ThinkingBlock | ToolUseBlock |
|------|-----------|--------------|-------------|
| `hasattr(block, "text")` | ✅ 匹配 | ❌ 也匹配（bug） | ❌ 不匹配 |
| `isinstance(block, TextBlock)` | ✅ 匹配 | ✅ 不匹配 | ✅ 不匹配 |

**原则：需要特定类型时用 `isinstance`，需要通用能力时用 `hasattr`。**

#### C++ 想修改原始数据怎么办？

```cpp
// 默认：值传递（拷贝），不影响原始数据
void modify(vector<int> v) {
    v[0] = 999;   // 改的是副本
}

// 加 &：引用传递，修改原始数据
void modify(vector<int>& v) {
    v[0] = 999;   // 改的是原始数据
}
```

| 语言 | 默认 | 想改原始数据 | 不想改原始数据 |
|------|------|------------|---------------|
| **Python** | 传引用 | 直接改 | `.copy()` 或 `deepcopy()` |
| **C++** | 传值 | 加 `&` | 什么都不加 |
