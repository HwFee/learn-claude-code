# s02 学习笔记 —— Tool Use

## 核心思想

> **"Adding a tool means adding one handler"**
> 
> 添加一个工具就是添加一个 handler。s02 的 `agent_loop` 和 s01 **一模一样**，只是多了工具和分发映射。

---

## 新增内容（相比 s01）

### 1. `safe_path` —— 路径沙箱

```python
def safe_path(p: str) -> Path:
    path = (WORKDIR / p).resolve()
    if not path.is_relative_to(WORKDIR):
        raise ValueError(f"Path escapes workspace: {p}")
    return path
```

- **作用**：防止**路径穿越攻击**（Path Traversal）
- 比如模型传入 `../../Windows/System32/hosts`，`resolve()` 后会变成绝对路径，然后 `is_relative_to(WORKDIR)` 发现它不在工作区内，直接抛异常
- s01 没有文件操作所以不需要，s02 有 read/write/edit 必须加这个限制

### 2. 四个工具函数

| 工具 | 功能 | 安全机制 |
|------|------|---------|
| `run_bash` | 执行 shell 命令 | 危险命令过滤 + `cwd=WORKDIR` |
| `run_read` | 读取文件内容 | `safe_path` + `limit` 行数限制 + 50000 字符截断 |
| `run_write` | 写入文件 | `safe_path` + `mkdir(parents=True)` 自动创建目录 |
| `run_edit` | 编辑文件（字符串替换） | `safe_path` + 检查 `old_text` 是否存在 + 只替换第一次 |

**`run_edit` 的设计**：用字符串匹配而不是行号，因为行号容易漂移（前面插入一行后面全乱）。要求 `old_text` 精确唯一，且 `replace(..., 1)` 只替换第一次出现。

### 3. `TOOL_HANDLERS` —— 分发映射

```python
TOOL_HANDLERS = {
    "bash":       lambda **kw: run_bash(kw["command"]),
    "read_file":  lambda **kw: run_read(kw["path"], kw.get("limit")),
    "write_file": lambda **kw: run_write(kw["path"], kw["content"]),
    "edit_file":  lambda **kw: run_edit(kw["path"], kw["old_text"], kw["new_text"]),
}
```

- 字典映射：工具名 → 处理函数
- `lambda **kw` 是**适配层**：统一接收 `block.input` 字典，取出对应字段传给底层函数
- `kw.get("limit")` vs `kw["command"]`：可选字段用 `get`，必填字段用 `[]`

### 4. `agent_loop` 的唯一变化

```python
# s01：硬编码
output = run_bash(block.input["command"])

# s02：通过 dispatch map 分发
handler = TOOL_HANDLERS.get(block.name)
output = handler(**block.input) if handler else f"Unknown tool: {block.name}"
```

加新工具时，**不需要改 `agent_loop`**，只需要：
1. 写 `run_xxx` 函数
2. 在 `TOOL_HANDLERS` 里加一行
3. 在 `TOOLS` schema 里加一行

### 5. `Path` 对象替代 `os.path`

- `WORKDIR = Path.cwd()` 替代 `os.getcwd()`
- `path.resolve()`、`path.is_relative_to()`、`path.mkdir(parents=True)`、`path.read_text()`、`path.write_text()` —— 比 `os.path` 更简洁

## 完整交互流程示例

用户输入：`帮我创建一个叫 hello.py 的文件，内容打印 Hello World`

模型可能会这样调用工具：

1. **第一轮**：`stop_reason='tool_use'`
   - `ToolUseBlock(name='write_file', input={'path': 'hello.py', 'content': 'print("Hello World")'})`
   - s02 执行 `run_write`，文件创建成功
   - 返回 `tool_result` 给模型

2. **第二轮**：`stop_reason='end_turn'`
   - 模型看到写入成功，回复用户 `已创建 hello.py`

或者模型可能先用 `bash` 看看当前目录有什么，再用 `write_file` —— 具体怎么调用由模型自己决定，harness 只负责执行。

## 附录：分发语法的从零讲解

如果你不熟悉 Python 的高级语法，下面逐行拆解 `TOOL_HANDLERS` 的每个部分。

### 1. `lambda` 是什么？

`lambda` 是 Python 里创建**匿名函数**的写法。

```python
# 正常函数
def add(a, b):
    return a + b

# lambda 等价写法
add = lambda a, b: a + b

# 调用方式完全一样
print(add(1, 2))  # 都输出 3
```

`lambda` 的特点：
- 没有 `def` 和函数名
- 只能写一行表达式
- 常用于

- 常用于"临时用一下"的场景，比如当参数传递、或者放在字典里

### 2. `**kw` 是什么？

`**kw` 是 Python 的**关键字参数解包**语法。

先看正常的函数参数：

```python
def greet(name, age):
    print(f"Hello {name}, you are {age}")

greet("Alice", 25)           # 位置传参
greet(name="Alice", age=25)  # 关键字传参
```

如果参数数量不固定，可以用 `**kw` 接收所有关键字参数：

```python
def my_func(**kw):
    print(kw)  # kw 是一个字典！

my_func(name="Alice", age=25, city="Beijing")
# 输出: {'name': 'Alice', 'age': 25, 'city': 'Beijing'}
```

`**kw` 的本质：**把所有 `key=value` 的参数，打包成一个字典 `kw`**。

反过来，`**字典` 可以把字典解包成关键字参数：

```python
def greet(name, age):
    print(f"Hello {name}, you are {age}")

info = {"name": "Alice", "age": 25}
greet(**info)  # 等价于 greet(name="Alice", age=25)
```

### 3. 为什么用 `lambda **kw`？不能直接传 `run_bash` 吗？

**不能，因为参数签名不匹配。**

看看四个函数的签名：

```python
run_bash(command: str) -> str               # 1个参数
run_read(path: str, limit: int = None) -> str   # 1个必填 + 1个可选
run_write(path: str, content: str) -> str       # 2个必填
run_edit(path: str, old_text: str, new_text: str) -> str  # 3个必填
```

它们参数数量和名字都不一样！

模型调用工具时，传过来的数据是这样的（一个字典）：

```python
block.name = "bash"
block.input = {"command": "echo hello"}  # 字典

block.name = "write_file"
block.input = {"path": "a.txt", "content": "hello"}  # 字典
```

**agent_loop 里的调用代码是统一的：**

```python
handler = TOOL_HANDLERS.get(block.name)
output = handler(**block.input)
```

这里 `handler(**block.input)` 等价于：
- 如果是 bash：`handler(command="echo hello")`
- 如果是 write_file：`handler(path="a.txt", content="hello")`

所以 `handler` 这个函数必须能接受**任意关键字参数**，然后自己在内部取出需要的字段。

### 4. `lambda **kw` 的真实作用

```python
"bash": lambda **kw: run_bash(kw["command"]),
```

拆解：
1. `lambda **kw`：定义一个匿名函数，接收任意关键字参数，打包成字典 `kw`
2. `kw["command"]`：从字典里取出 `"command"` 这个键的值
3. `run_bash(kw["command"])`：把这个值传给真正的 `run_bash` 函数

等价于：

```python
def bash_handler(**kw):
    return run_bash(kw["command"])

"bash": bash_handler,
```

用 `lambda` 只是更简洁，不需要起名字。

### 5. `kw["key"]` vs `kw.get("key")` 的区别

```python
"read_file": lambda **kw: run_read(kw["path"], kw.get("limit")),
```

- `kw["path"]`：**必填字段**。如果字典里没有 `"path"`，直接抛 `KeyError` 报错。
- `kw.get("limit")`：**可选字段**。如果字典里没有 `"limit"`，返回 `None`（不会报错）。

对应 `run_read(path: str, limit: int = None)` 的签名：`path` 必填，`limit` 可选，不传就是 `None`。

### 6. 完整分发流程走一遍

假设模型要调用 `write_file`：

**Step 1: 模型返回 ToolUseBlock**

```python
block.name = "write_file"
block.id = "call_00_xxx"
block.input = {"path": "hello.py", "content": "print(1)"}
```

**Step 2: agent_loop 查找 handler**

```python
handler = TOOL_HANDLERS.get("write_file")
# handler 现在等于: lambda **kw: run_write(kw["path"], kw["content"])
```

**Step 3: 解包调用**

```python
output = handler(**block.input)
# 等价于: handler(path="hello.py", content="print(1)")
# lambda 内部: run_write(kw["path"], kw["content"])
# 最终调用: run_write("hello.py", "print(1)")
```

**Step 4: 组装 tool_result**

```python
results.append({
    "type": "tool_result",
    "tool_use_id": "call_00_xxx",   # 必须和 block.id 对应
    "content": "Wrote 9 bytes to hello.py"
})
```

### 7. 如果不用 lambda，还能怎么写？

**写法A：不用 lambda，直接写普通函数**

```python
def handle_bash(**kw):
    return run_bash(kw["command"])

def handle_read(**kw):
    return run_read(kw["path"], kw.get("limit"))

TOOL_HANDLERS = {
    "bash": handle_bash,
    "read_file": handle_read,
}
```

功能完全一样，就是代码变长了。

**写法B：不用 **kw，用 functools.partial**

```python
from functools import partial

TOOL_HANDLERS = {
    "bash": partial(run_bash),  # 不太好用，因为参数名不统一
}
```

`partial` 适合参数固定的场景，这里每个工具参数不同，不如 `lambda **kw` 灵活。

### 8. `**kw` 的额外好处：多传少传都不报错

```python
# lambda **kw 的 handler：多传参数也没事
d = {"path": "a.txt", "content": "hello", "extra": "xxx"}
handler(**d)   # lambda 内部用 kw.get 取需要的，ignore 多余的

# 但如果直接把字典当参数传：
handler(d)     # 报错！run_write 期望的是 path=..., content=...，不是字典
```

`lambda **kw` 做了一层"缓冲"：不管模型传了多少字段，handler 都照单全收，然后自己挑需要的用。

---

**写法C：不用字典分发，用 if-elif**

```python
if block.name == "bash":
    output = run_bash(block.input["command"])
elif block.name == "read_file":
    output = run_read(block.input["path"], block.input.get("limit"))
elif block.name == "write_file":
    output = run_write(block.input["path"], block.input["content"])
# ...
```

这是 s01 的写法。s02 用字典就是为了**避免这种硬编码的 if-elif 链**，加新工具时只改字典不用改这段逻辑。
