# Message 对象结构解析

## 顶层字段

| 字段              | 类型            | 说明                               |
| --------------- | ------------- | -------------------------------- |
| `id`            | `str`         | 消息唯一标识符，每次 API 调用生成一个新的          |
| `container`     | `None`        | 容器信息，一般不用                        |
| `content`       | `list[Block]` | **核心**：内容块列表，可能包含多种类型的块          |
| `model`         | `str`         | 实际使用的模型名，这里是 `deepseek-v4-flash` |
| `role`          | `str`         | 固定为 `"assistant"`                |
| `stop_reason`   | `str`         | 模型停止原因，见下表                       |
| `stop_sequence` | `str \| None` | 如果使用 stop sequence 触发停止，这里是序列值   |
| `type`          | `str`         | 固定为 `"message"`                  |
| `usage`         | `Usage`       | Token 用量统计对象                     |

## stop_reason 枚举值

| 值 | 含义 |
|----|------|
| `"end_turn"` | 模型正常说完话了，可以返回给用户 |
| `"tool_use"` | 模型要调用工具，需要执行后把结果喂回去 |
| `"max_tokens"` | 达到 max_tokens 限制被截断，可能需要续写 |

## content 中的 Block 类型

`content` 是一个列表，Claude 支持在一次回复中返回多种类型的内容块：

| Block 类型 | `type` 字段 | 关键属性 | 出现场景 |
|-----------|------------|---------|---------|
| `TextBlock` | `"text"` | `text: str` | 模型直接回复文本 |
| `ThinkingBlock` | `"thinking"` | `thinking: str`, `signature: str` | 扩展思考模式（Claude 3.7 Sonnet / DeepSeek 等） |
| `ToolUseBlock` | `"tool_use"` | `name: str`, `input: dict`, `id: str` | 模型请求调用工具 |

> 本次返回了 `ThinkingBlock`，说明 `deepseek-v4-flash` 支持 thinking/reasoning 模式，thinking 内容是模型的内部推理过程。

## Usage 对象字段

| 字段 | 说明 |
|------|------|
| `input_tokens` | 本次请求消耗的输入 token 数（46） |
| `output_tokens` | 模型生成的输出 token 数（114） |
| `cache_read_input_tokens` | 从 prompt cache 中读取的 token 数（256），省钱 |
| `cache_creation_input_tokens` | 新创建 cache 的 token 数（0） |

## 为什么 content 是列表而不是字符串？

这是 Anthropic API 的重要设计：

1. **文本 + 工具调用可以混排**：模型可以先说一段话，然后调用工具
2. **支持并行工具调用**：一次返回多个 `ToolUseBlock`，同时调用多个工具
3. **支持扩展思考**：`ThinkingBlock` 和 `TextBlock` 可以分开

在 s01 的代码里，这段逻辑就是处理 content 列表：

```python
for block in response.content:
    if block.type == "tool_use":      # 执行工具
        ...
    elif block.type == "text":        # 打印文本
        print(block.text)
```

## 实际使用注意点

- **不要假设 content 只有一个块**：即使看起来只有文本，也要遍历列表
- **ThinkingBlock 通常不需要展示给用户**：它是模型的内部思考过程
- **`tool_use_id` 必须和 `tool_result` 配对**：API 要求一一对应

# s01 循环实战：执行 `echo Hello World`

## 用户输入

```
s01 >> 你执行一下工具 在控制台打印一个Hello World
```

## 完整交互流程

这个交互展示了 s01 核心循环的**两次 API 调用**：

### 第一轮：模型决定调用工具（stop_reason='tool_use'）

```python
Message(
    id='3eb21bc4-aa97-4f0a-a4cc-4363f8740481',
    content=[
        ThinkingBlock(
            thinking='用户要求我执行一个工具，在控制台打印"Hello World"。
                     我可以使用bash工具来执行这个命令。
                     在Windows上，我可以用`echo Hello World`命令。',
            type='thinking'
        ),
        ToolUseBlock(
            id='call_00_YrpzpSsrsZujp64kZtfx3315',
            input={'command': 'echo Hello World'},
            name='bash',
            type='tool_use'
        )
    ],
    stop_reason='tool_use',
    usage=Usage(input_tokens=143, output_tokens=82, cache_read_input_tokens=256)
)
```

**发生了什么：**
1. 用户的输入被包装成 `{"role": "user", "content": "你执行一下工具..."}` 加入 messages
2. `agent_loop` 第一次调用 `client.messages.create()`
3. 模型分析任务：需要用 bash 执行 `echo Hello World`
4. 模型返回 `stop_reason='tool_use'`，表示要调用工具
5. `content` 里有两个块：
   - `ThinkingBlock`：模型的内部思考过程
   - `ToolUseBlock`：实际的工具调用请求，包含 `id`、`name='bash'`、`input={'command': 'echo Hello World'}`

**s01 代码的执行路径：**
```python
# response.stop_reason == "tool_use"，不 return，继续执行
for block in response.content:
    if block.type == "tool_use":
        print(f"\033[33m$ {block.input['command']}\033[0m")   # 黄色打印: $ echo Hello World
        output = run_bash(block.input["command"])            # 执行命令
        print(output[:200])                                   # 打印: Hello World
        results.append({
            "type": "tool_result",
            "tool_use_id": block.id,                          # "call_00_YrpzpSsrsZujp64kZtfx3315"
            "content": output                                  # "Hello World"
        })
messages.append({"role": "user", "content": results})        # 把结果追加到对话
```

### 第二轮：模型正常回复（stop_reason='end_turn'）

```python
Message(
    id='c3aae16d-a566-4a1c-a661-0356c6600db4',
    content=[
        ThinkingBlock(
            thinking='成功打印了"Hello World"。',
            type='thinking'
        ),
        TextBlock(
            text='成功在控制台打印了 **Hello World**！🎉',
            type='text'
        )
    ],
    stop_reason='end_turn',
    usage=Usage(input_tokens=111, output_tokens=21, cache_read_input_tokens=384)
)
```

**发生了什么：**
1. `agent_loop` 再次调用 `client.messages.create()`，这次 messages 里多了 tool_result
2. 模型看到工具执行成功，决定不再调用工具，直接回复用户
3. `stop_reason='end_turn'`，循环结束，`agent_loop` 返回
4. `content` 里有两个块：
   - `ThinkingBlock`：模型确认任务完成
   - `TextBlock`：最终回复文本

**s01 代码的执行路径：**
```python
# response.stop_reason == "end_turn"，直接 return
if response.stop_reason != "tool_use":
    return
```

## Q4：`hasattr` 是什么？

**定义：** `hasattr(obj, name)` 是 Python 内置函数，检查对象 `obj` 是否拥有名为 `name` 的属性或方法，返回 `True` 或 `False`。

**在 s01 中的代码：**

```python
response_content = history[-1]["content"]
if isinstance(response_content, list):
    for block in response_content:
        if hasattr(block, "text"):      # ← 这里
            print(block.text)
```

**为什么要用 `hasattr`？**

`s01` 最后一条 message 的 `content` 是一个列表，里面可能包含不同类型的 block：

| block 类型 | 是否有 `.text` | 示例 |
|-----------|---------------|------|
| `TextBlock` | ✅ 有 | `TextBlock(text='Hello', type='text')` |
| `ToolUseBlock` | ❌ 没有 | `ToolUseBlock(name='bash', input={...}, type='tool_use')` |
| `ThinkingBlock` | ❌ 没有 | `ThinkingBlock(thinking='...', type='thinking')` |

如果不判断直接写 `print(block.text)`：
- 遇到 `TextBlock` → 正常打印
- 遇到 `ToolUseBlock` → 抛 `AttributeError: 'ToolUseBlock' object has no attribute 'text'`

**不用 `hasattr` 的替代写法：**

```python
# 写法1：直接用 hasattr（最简洁）
if hasattr(block, "text"):
    print(block.text)

# 写法2：用 getattr 提供默认值
print(getattr(block, "text", ""))

# 写法3：用 isinstance 判断类型（更明确）
from anthropic.types import TextBlock
if isinstance(block, TextBlock):
    print(block.text)

# 写法4：try/except（不推荐，EAFP 风格）
try:
    print(block.text)
except AttributeError:
    pass
```

s01 选择了 `hasattr`，因为代码简洁且不需要 import 具体类型。

## 关键结论

1. **`stop_reason='tool_use'` 时循环继续**：s01 执行工具，把结果喂回去
2. **`stop_reason='end_turn'` 时循环结束**：模型直接回复，agent_loop 返回
3. **ToolUseBlock.id 必须和 tool_result.tool_use_id 配对**：API 内部用这个 id 关联调用和结果
4. **两次 API 调用，同一个 `agent_loop` 函数**：代码没有变，是 messages 的内容在变
5. **ThinkingBlock 是内部思考**：用户看到的是 `TextBlock.text`，thinking 内容对用户不可见

## Q5：为什么 tool_result 的 role 是 `"user"`？

在 s01 的代码里，工具执行结果是这样追加到 messages 的：

```python
messages.append({"role": "user", "content": results})
```

**问题来了：工具结果是模型自己调用的，为什么 role 不是 `"assistant"` 而是 `"user"`？**

### 原因1：API 要求消息必须交替

Claude API 的消息格式必须是 **user → assistant → user → assistant** 交替出现，不能连续两个 assistant：

```python
# ❌ 错误：连续两个 assistant
[
    {"role": "user", "content": "帮我查一下天气"},
    {"role": "assistant", "content": [ToolUseBlock(...)]},   # assistant 调工具
    {"role": "assistant", "content": "结果是晴天"},           # ❌ 又 assistant！
]

# ✅ 正确：交替
[
    {"role": "user", "content": "帮我查一下天气"},
    {"role": "assistant", "content": [ToolUseBlock(...)]},   # assistant 调工具
    {"role": "user", "content": [tool_result...]},            # ✅ user 收到结果
    {"role": "assistant", "content": "结果是晴天"},           # ✅ assistant 回复
]
```

### 原因2：从模型的视角，tool_result 是"外部输入"

想象这个对话：

| role | 内容 | 谁说的 |
|------|------|--------|
| user | "帮我查一下天气" | 人类用户 |
| assistant | "我要调用天气工具" | 模型决定调用工具 |
| **user** | "工具返回：晴天 25°C" | **外部世界给模型的信息** |
| assistant | "今天晴天，25度" | 模型基于信息回复 |

**tool_result 对模型来说，和用户的输入是一样的** —— 都是"别人告诉我的信息"。模型自己是 `"assistant"`，它接收到的所有外部信息（无论是用户打字还是工具返回）都是 `"user"`。

### 原因3：Anthropic API 的协议设计

Anthropic 故意这样设计：
- `"user"` = 任何**输入给模型**的信息
- `"assistant"` = 模型**自己生成**的信息

这保持了对话结构的一致性，也让模型无法区分"这是用户说的话"和"这是工具返回的结果"——对模型来说，都是它要基于这些信息进行推理的上下文。

### 总结

```
用户提问    →  role="user"      →  模型接收信息
模型调工具  →  role="assistant"  →  模型主动行动
工具返回    →  role="user"      →  模型接收信息（工具是外部世界）
模型回复    →  role="assistant"  →  模型生成内容
```

**工具结果用 `"user"` 角色，是因为它代表"外部世界给模型的输入"，而不是"模型自己说的话"。**
