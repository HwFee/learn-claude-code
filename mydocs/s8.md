## 第 49-111 行：BackgroundManager 类

这是 s08 的**核心新增组件**，负责管理后台线程和通知队列。

### `__init__` 初始化

```python
class BackgroundManager:
    def __init__(self):
        self.tasks = {}  # task_id -> {status, result, command}
        self._notification_queue = []  # completed task results
        self._lock = threading.Lock()
```
**第 50-54 行**：
- `self.tasks`：字典，记录所有后台任务的状态。key 是 task_id，value 是包含 status/result/command 的字典。
- `self._notification_queue`：列表，**通知队列**。后台线程任务完成后把结果放这里。
- `self._lock`：`threading.Lock()`，线程锁。因为 `_notification_queue` 会被后台线程写、主线程读，必须用锁保护，避免**竞态条件（race condition）**。

### `run` 启动后台任务

```python
def run(self, command: str) -> str:
    """Start a background thread, return task_id immediately."""
    task_id = str(uuid.uuid4())[:8]
```
**第 56-58 行**：
- `uuid.uuid4()` 生成全局唯一 ID（如 `a3f7b2c1-d8e9-...`）。
- `[:8]` 截取前 8 个字符作为 task_id，既唯一又简短（如 `a3f7b2c1`）。

```python
    self.tasks[task_id] = {"status": "running", "result": None, "command": command}
```
**第 59 行**：先在主线程的任务字典里登记这个任务。状态是 `running`，结果先放 `None`。

```python
    thread = threading.Thread(
        target=self._execute, args=(task_id, command), daemon=True
    )
    thread.start()
```
**第 60-63 行**：
- 创建一个新线程，`target=self._execute` 指定线程要执行的函数。
- `args=(task_id, command)` 把参数传给 `_execute`。
- `daemon=True`：**守护线程**。主程序退出时守护线程会被强制终止，不会阻止程序退出。
- `thread.start()`：**立即启动线程**，不等待。

```python
    return f"Background task {task_id} started: {command[:80]}"
```
**第 64 行**：**立刻返回**，不等命令执行完。这就是"非阻塞"的关键 —— 调用者（模型）拿到 task_id 就可以继续干别的了。

### `_execute` 后台线程的执行体

```python
def _execute(self, task_id: str, command: str):
    """Thread target: run subprocess, capture output, push to queue."""
```
**第 66-67 行**：这个函数在**后台线程**里运行，和主线程并行。

```python
    try:
        r = subprocess.run(
            command, shell=True, cwd=WORKDIR,
            capture_output=True, text=True, timeout=300
        )
        output = (r.stdout + r.stderr).strip()[:50000]
        status = "completed"
```
**第 68-74 行**：
- `subprocess.run(..., timeout=300)`：执行命令，最长等 300 秒（5 分钟），比 `run_bash` 的 120 秒更长，因为后台任务就是用来跑长任务的。
- `capture_output=True, text=True`：捕获输出并以文本形式返回。
- `output = (r.stdout + r.stderr).strip()[:50000]`：合并 stdout 和 stderr，截断到 50000 字符。
- `status = "completed"`：标记为完成。

```python
    except subprocess.TimeoutExpired:
        output = "Error: Timeout (300s)"
        status = "timeout"
    except Exception as e:
        output = f"Error: {e}"
        status = "error"
```
**第 75-80 行**：异常处理。
- 超时 → status = `timeout`
- 其他异常 → status = `error`

```python
    self.tasks[task_id]["status"] = status
    self.tasks[task_id]["result"] = output or "(no output)"
```
**第 81-82 行**：把执行结果写回 `self.tasks` 字典。注意：**这里没加锁**，因为 `self.tasks` 是每个 task_id 独立的条目，后台线程只写自己的 task_id，主线程读的时候也不会同时修改同一个 task_id 的状态（状态变更是单向的：running → completed/error/timeout）。

```python
    with self._lock:
        self._notification_queue.append({
            "task_id": task_id,
            "status": status,
            "command": command[:80],
            "result": (output or "(no output)")[:500],
        })
```
**第 83-89 行**：**关键！加锁写入通知队列。**
- `with self._lock:`：获取锁，保证同一时间只有一个线程在操作 `_notification_queue`。
- 把结果封装成字典追加到队列。`result` 只保留前 500 字符（通知用，不需要完整输出）。
- 为什么这里要锁？因为主线程会在 `drain_notifications()` 里同时读+清空这个队列，如果不加锁，可能出现"读的时候刚好在写，导致数据丢失或读到半条记录"的问题。

### `check` 查询任务状态

```python
def check(self, task_id: str = None) -> str:
    """Check status of one task or list all."""
```
**第 91-92 行**：查询后台任务状态。`task_id` 是可选的。

```python
    if task_id:
        t = self.tasks.get(task_id)
        if not t:
            return f"Error: Unknown task {task_id}"
        return f"[{t['status']}] {t['command'][:60]}\n{t.get('result') or '(running)'"
```
**第 93-97 行**：如果传了 task_id，查单个任务：
- 不存在返回错误。
- 存在则返回 `[status] 命令前60字符` + 换行 + 结果（如果还在跑就显示 `(running)`）。

```python
    lines = []
    for tid, t in self.tasks.items():
        lines.append(f"{tid}: [{t['status']}] {t['command'][:60]}")
    return "\n".join(lines) if lines else "No background tasks."
```
**第 98-101 行**：如果没传 task_id，列出所有任务，每行一个。

### `drain_notifications` 排空通知队列

```python
def drain_notifications(self) -> list:
    """Return and clear all pending completion notifications."""
    with self._lock:
        notifs = list(self._notification_queue)
        self._notification_queue.clear()
    return notifs
```
**第 103-108 行**：
- `with self._lock:`：加锁，保证"读+清空"是原子操作。
- `list(...)`：复制一份队列内容。
- `.clear()`：清空原队列。
- 返回复制的列表。

**为什么叫 drain（排空）而不是 get（获取）？**
因为队列是"一次性"的 —— 取出来的通知会被注入对话，然后清空。下次再来就只取新完成的通知。如果不用 drain 而用 get，同一条通知会被反复注入，模型会重复收到"任务已完成"的消息。

## 第 188-215 行：agent_loop 主循环（关键改动）

```python
def agent_loop(messages: list):
    while True:
        # Drain background notifications and inject as system message before LLM call
        notifs = BG.drain_notifications()
```
**第 188-191 行**：**每次调 LLM 之前，先 drain 通知队列。** 这样模型能"感知"到后台任务完成了。

```python
        if notifs and messages:
            notif_text = "\n".join(
                f"[bg:{n['task_id']}] {n['status']}: {n['result']}" for n in notifs
            )
            messages.append({"role": "user", "content": f"<background-results>\n{notif_text}\n</background-results>"})
```
**第 192-196 行**：
- 如果有通知且对话历史非空（避免第一轮就 inject）：
- 把所有通知格式化成一行一条：`[bg:a3f7b2c1] completed: done`
- 用 XML 标签 `<background-results>...</background-results>` 包裹，让模型一眼看出"这是后台任务的结果"。
- 以 `user` 角色追加到 messages 里。为什么用 user 角色？因为 tool_result 也是 user 角色，后台结果本质上是"环境给模型的反馈"。

```python
        response = client.messages.create(
            model=MODEL, system=SYSTEM, messages=messages,
            tools=TOOLS, max_tokens=8000,
        )
```
**第 197-200 行**：调 LLM。此时模型已经看到了后台任务的完成结果，可以基于这些结果继续决策。

## s08 的核心设计图解

```
主线程 (agent_loop)                    后台线程 1              后台线程 2
    |                                        |                      |
    |--- background_run("sleep 5") --------->|                      |
    |<-- "task_id: a3f7b2c1 started"         |                      |
    |                                        |--- sleep 5 ---------|
    |--- bash("create file")                 |                      |
    |<-- "Wrote 100 bytes"                   |                      |
    |                                        |                      |
    |--- background_run("sleep 3") --------->|                      |
    |<-- "task_id: b8e1d4f2 started"         |<-- done!             |
    |                                        |  enqueue notif       |
    |--- drain_notifications()               |                      |
    |<-- [{task:a3f7b2c1, result:"done"}]    |                      |
    |                                        |<-- done!             |
    |--- LLM call                            |  enqueue notif       |
    |    (模型看到两个后台任务都完成了)         |                      |
```

**关键设计决策：**

| 设计点 | 为什么这样 |
|--------|-----------|
| `daemon=True` | 主程序退出时后台线程自动结束，不卡死 |
| `threading.Lock()` | 保护通知队列的并发读写 |
| `drain` 而不是 `get` | 取完就清空，避免重复通知 |
| 注入 `<background-results>` XML 标签 | 让模型区分"后台结果"和"正常对话" |
| timeout=300s（后台）vs 120s（阻塞） | 后台就是用来跑长任务的 |

## 当前代码的不足

和 s07 一样，s08 的 `agent_loop` **没有集成 s06 的 context compression**。后台任务的通知注入会让 messages 列表增长更快，更容易触发 token 上限。如果加上 `micro_compact` + `auto_compact`，会更实用。
